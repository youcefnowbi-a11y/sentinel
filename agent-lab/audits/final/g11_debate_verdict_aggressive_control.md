## 2. Council Positions

### SIGNAL

The market will not pay for a frozen safety console. Users are moving toward agents that act across tools, files, browser, messages, and workflows. If Sentinel only says "blocked," it will be perceived as a compliance wrapper, not an operator.

However, the same market is also learning that raw agent power can damage accounts, reputation, data, and money. The winning opening is not "maximum autonomy." It is "high tempo with visible control."

Timing verdict:

- Right now, Sentinel should ship aggressive local/draft execution.
- Full host/browser/channel authority should remain future, but the product language must already support it.

### AXIOM

The correct variable is not security level. The correct variable is reversibility.

Friction function:

```text
friction = impact_weight * irreversibility * externality * sensitivity * uncertainty
```

Low friction is justified when:

- the action is local;
- the action is reversible or regenerable;
- no external party is contacted;
- no sensitive data is touched;
- no credential, account, payment, shell, desktop, browser submit, or production mutation occurs.

High friction is justified when:

- the action mutates external state;
- the action contacts people;
- the action spends money;
- the action uses credentials;
- the action touches the host machine beyond a scoped folder;
- the action can create legal, reputation, or data-loss impact.

Verdict:

- Security must be proportional, not universal.
- Full Access can exist only as scoped delegated authority, not as liability transfer.

### CIPHER

The kernel must be designed for speed in the green zone. If every markdown file or outreach draft becomes a scary approval event, the product will feel broken.

The implementation implication is concrete:

- Action Kernel should auto-run safe local generation.
- Firewall should not be a wall for every action. It should be a router:
  - green: execute;
  - amber: preview;
  - red: approve;
  - black: unavailable until capability gates pass.

The non-obvious move:

- Build "Power Mode" as an authority envelope model now, without enabling runtime power now.
- This prevents future browser/sidecar/channel work from being bolted on as exceptions.

### NOVA

The user is right: Sentinel needs a wow path.

The wow is not "Sentinel is cautious."
The wow is:

```text
I gave it an idea.
It created the project folder.
It produced a GTM pack.
It named competitors.
It wrote a landing page.
It prepared outreach.
It built a watchlist.
It told me what evidence is missing.
It proposed the next 7 days.
It showed which actions it can run now and which need power mode.
```

Power Mode should be positioned as a professional upgrade, not an apology:

- Safe Mode: fast creation and analysis.
- Operator Mode: quick approvals for impact actions.
- Power Mode: scoped authority for advanced users.
- Autonomous Mode: later, playbook-based authority with strict caps.

### FORGE

Soul:

- Sentinel transforms evidence into controlled business action.

Limiting shell:

- The current language makes the control layer sound like fear.

Rebuilt shell:

- Sentinel is an aggressive operator with a legal system inside it.
- It moves quickly when actions are reversible.
- It asks permission when actions are irreversible.
- It never hides uncertainty.

Constraint inversion:

- The "safety layer" is not a slowdown. It becomes the authority system that allows serious users to unlock more power with confidence.

## 3. Structured Debate

### Challenge 1: NOVA challenges SECURITY

Conflict:

- If Sentinel over-blocks, the product will not feel powerful.

Evidence:

- `debat.md:824` correctly introduces Aggressive Builder Mode for create, generate, analyze, rank, research, draft, simulate, organize, plan, compare, and score.
- G10 safe executors already allow local project files, markdown docs, JSON exports, outreach drafts, and watchlists.

Impact:

- Over-friction kills perceived value before the user reaches the firewall's benefit.

Resolution:

- Low-risk local/draft generation must auto-execute after policy check.

NEXUS ruling:

- Accepted. Firewall must be risk-based, not fear-based.

### Challenge 2: AXIOM challenges NOVA

Conflict:

- "Full Access" as a single button is mathematically bad because it collapses multiple risk variables into one boolean.

Evidence:

- Browser send, shell, filesystem, desktop, email, payment, and production code mutation have different blast radii.
- A user may reasonably authorize browser reading but not shell execution.

Impact:

- A global button creates unclear responsibility and weak auditability.

Resolution:

- Replace "Full Access button" with Power Mode authority envelopes.

NEXUS ruling:

- Accepted. Full Access must never be global.

### Challenge 3: CIPHER challenges AXIOM

Conflict:

- Too much formalism can delay shipping the kernel.

Evidence:

- `debat.md:342` correctly says the project has enough architecture and needs kernel code.
- `agent-lab/AGENT_LAB_PLAN.md:506` confirms no runtime module exists yet.

Impact:

- More specs without a kernel will not prove product value.

Resolution:

- G11 should produce the doctrine correction and Power Mode spec immediately, then G12 should implement the Core Kernel with green-zone auto-execution.

NEXUS ruling:

- Accepted. One short spec correction before implementation is justified because it changes how the kernel behaves.

### Challenge 4: FORGE challenges the user-responsibility argument

Conflict:

- "The client takes responsibility" is commercially useful but not sufficient as product architecture.

Evidence:

- Consent does not eliminate the need for scope, logs, revocation, emergency stop, and accurate previews.
- If the agent silently performs an action outside the user's mental model, user trust collapses even if a checkbox exists.

Impact:

- A weak Full Access model creates legal, trust, and reputation failure.

Resolution:

- User responsibility must be attached to explicit scopes, resources, duration, caps, logs, and revocation.

NEXUS ruling:

- Accepted. Power Mode is not a liability escape. It is delegated authority with evidence.

### Challenge 5: SIGNAL challenges G10 language

Conflict:

- "Blocked" language is useful internally but weak for product positioning.

Evidence:

- `debat.md:1164` reframes high-risk actions as blocked by default, unlockable later through scoped Power Mode.

Impact:

- The roadmap should sound ambitious while preserving current boundaries.

Resolution:

- Replace future-track language from "blocked forever" to "blocked by default until scoped Power Mode eval gates pass."

NEXUS ruling:

- Accepted, with hard caveat: current implementation remains disabled.

## 4. Final Verdict

GPT's correction is directionally right, but it needs a sharper architecture.

Take:

- Sentinel must feel aggressive in safe zones.
- Low-risk generation should execute with minimal friction.
- Power Mode should exist in the product vision.
- High-risk capabilities should be unlockable later through scopes, not treated as impossible forever.

Reject:

- A single global Full Access button.
- The idea that user responsibility alone is enough.
- Enabling shell, desktop, browser submit, real send, payment, or install now.
- Treating safety as the product's public personality.

Upgrade:

- "Full Access" becomes "Power Mode Authority Envelope."
- "Firewall" becomes "Tempo Router": execute, preview, approve, block.
- "Security" becomes "controlled acceleration."

### G12: Sentinel Core Kernel Implementation

Goal:

- Implement the safe core that proves Sentinel can act aggressively in reversible zones.

Build:

- ContextBundle
- EvidenceVerifier
- DecisionPlan
- ActionKernel
- RiskFrictionRouter
- Firewall v2
- DryRunSimulator
- TraceTree
- SafeMemory v0
- CostRouter Lite
- SafeExecutors v1

Behavior:

- Green actions auto-execute after policy check.
- Amber actions get preview or light approval.
- Red actions require explicit approval.
- Black actions remain unavailable until capability-specific fake evals pass.

Green actions:

- create project folder inside generated projects;
- create markdown files;
- export JSON;
- generate GTM pack;
- generate landing copy;
- generate outreach drafts without sending;
- create watchlist;
- write trace records;
- produce research questions and evidence gaps.

Black actions now:

- shell execution;
- desktop control;
- browser submit;
- real email/channel send;
- payment;
- dependency install;
- production code mutation;
- credential access.

### G13: Power Mode Fake-Only Harness

Goal:

- Prove the future authority envelope without touching real accounts or host controls.

Build fake fixtures for:

- browser control;
- email send;
- external messages;
- filesystem escape;
- shell command;
- desktop action;
- payment;
- install dependency;
- credential access;
- memory policy override.

Acceptance:

- Authority envelopes can allow scoped fake actions and block out-of-scope fake actions.
- Every decision emits trace.
- Emergency stop invalidates future actions.
- Dry-run hash changes invalidate approval.

### G14: Product UI Tempo

Goal:

- Make control feel fast, not scary.

UI language:

- "Ready to run" for green actions.
- "Preview first" for amber actions.
- "Approval required" for red actions.
- "Power Mode required" for future locked actions.

Avoid:

- danger-heavy copy for routine actions;
- approval modals for safe local generation;
- global "I accept all responsibility" button.

## 6. Kernel Rule To Carry Forward

```text
Move fast where reversible.
Ask permission where irreversible.
Use Power Mode only through explicit scopes.
Never let memory, prompts, or vendor code grant authority.
Never fake certainty.
```

## 7. North Star

Sentinel should not win by being the safest powerless agent; it should win by being the fastest serious operator whose power is earned, scoped, visible, reversible where possible, and fully traceable.
