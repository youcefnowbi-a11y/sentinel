# Brain Event Catalog

Date: 2026-04-28
Status: Core Brain Lock catalog

This catalog lists the current brain events and how they participate in replay,
audit, final gate certification, and evidence. Event payloads listed here are
the minimum contract observed in the current core. Future modules must update
this catalog before adding or changing event semantics.

## Event Rules

- Events are append-only.
- Event ids are unique inside one trace.
- Event hashes include canonical payload content.
- Mission ids must remain mono-mission across the trace.
- Phase transitions must be legal.
- Terminal events must be last.
- Events that shape planning, execution, repair, learning, or completion must
  be backed by evidence chains where required by the certification gate.

## Evidence Chain Requirements

The current evidence reviewer enforces these minimum relationships:

- `tool_selection` evidence before `plan_created` and terminal block when tool
  selection already occurred.
- `hypothesis_verdict` evidence before `plan_created` and terminal block when
  hypothesis review already occurred.
- `plan_creation` evidence before `worker_started` and terminal block when plan
  review already occurred.
- `repair_decision` evidence before `success_evaluated` and before escalation
  when a repair decision already occurred.
- `success_evaluation` evidence before `learning_proposed` when learning follows
  success evaluation.
- `learning_proposal` evidence before `agent_completed` or `agent_failed` when
  learning proposals were emitted.
- `browser_cortex_interpretation` evidence is optional but, when emitted, must
  reference browser trace events and must not create mission authority.

## Replay Snapshot Fields

Replay reconstructs the following directly from events:

- mission id, event count, final phase, last event id, trace hash;
- selected methods;
- execution posture, direct tool call budget, max repair cycles;
- needed and missing capabilities;
- selected, candidate, blocked, and unavailable tools;
- verified and rejected hypotheses;
- selected action id/name;
- effort level/score;
- project path;
- controlled capability executed/rejected counts;
- repair decision, pressure, cycles;
- success boolean;
- learning proposal count;
- evidence chain ids and decision types.

If a future event changes any of these fields, replay tests must be updated in
the same change.
